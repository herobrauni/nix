# NixOS Fleet Management Plan

## Overview

Manage all NixOS hosts from this single flake repo using the **den** (dendritic) pattern.
Key goals: cachix-backed remote builds, agenix-encrypted secrets, tmpfs impermanence,
and a migration path from existing Debian servers.

## 3. Den Configuration Pattern

### 3.1 Host Declarations (`hosts.nix`)

```nix
{
  den.hosts.x86_64-linux.aurora.users.brauni = {};
  den.hosts.x86_64-linux.atlas.users.brauni = {};
  den.hosts.x86_64-linux.laptop.users.brauni = {};
}
```

### 3.2 Host Aspects (`hosts/aurora.nix`)

Each host gets its own aspect file. Hosts include shared aspects via `includes`:

```nix
{ den, ... }:
{
  den.aspects.aurora = {
    includes = [
      den.aspects.base-server        # shared server config
      den.aspects.impermanence       # tmpfs impermanence
      den.aspects.cachix-client      # use cachix
    ];

    nixos = { pkgs, ... }: {
      networking.hostName = "aurora";
      # host-specific config here
    };
  };
}
```

### 3.3 Shared Aspects (`aspects/`)

Aspects are the key den pattern — they're composable, reusable configuration bundles
that can be included by any host or user:

```nix
# aspects/impermanence.nix
{ inputs, ... }:
{
  den.aspects.impermanence = {
    nixos = { pkgs, ... }: {
      imports = [ inputs.impermanence.nixosModules.impermanence ];
      environment.persistence."/persist" = {
        directories = [
          "/var/log"
          "/var/lib/nixos"
          "/etc/ssh"
        ];
        files = [
          "/etc/machine-id"
        ];
      };
      boot.initrd.postDeviceCommands = ''
        mkdir -p /mnt-root/persist
      '';
      fileSystems."/persist".neededForBoot = true;
    };
  };
}
```

### 3.4 User Aspects (`users/brauni.nix`)

```nix
{ den, ... }:
{
  den.aspects.brauni = {
    includes = [
      den.provides.define-user
      den.provides.primary-user
      (den.provides.user-shell "bash")  # or fish/zsh
    ];

    homeManager = { pkgs, ... }: {
      home.packages = with pkgs; [ git vim htop ];
    };
  };
}
```

## 4. Tmpfs Impermanence

**Pattern:** Root filesystem is `tmpfs` (wiped on every boot). Persistent data lives on a `/persist` mount.

### Implementation Strategy

1. **Root `/` is tmpfs** — configured via `fileSystems."/".fsType = "tmpfs"`
2. **`/persist` is a real filesystem** (ext4/btrfs/zfs on a separate partition or subvolume)
3. **`impermanence` module** declares what survives reboots:
   - System: `/var/log`, `/var/lib/nixos`, `/etc/ssh`, `/etc/machine-id`
   - Per-host: anything host-specific (e.g. `/var/lib/postgresql` for a DB server)
   - Per-user: `~/.local/share`, `~/.config`, `~/.ssh`, etc.
4. **Boot process**: initrd mounts `/persist` first, then tmpfs for `/`, then `impermanence` bind-mounts things back
5. **Opt-in approach**: nothing persists unless explicitly declared — forces conscious decisions about state

### Shared aspect: `aspects/impermanence.nix`
- Common persistence entries (system-wide)
- Host aspects extend with host-specific entries
- User home-manager persistence for dotfiles

## 5. Agenix Secrets Management

### How it works

1. **Each host generates an SSH host key** on first deploy → this becomes the decryption key
2. **`secrets.nix`** declares which secrets exist and which hosts can decrypt them
3. **Secrets are `.age` files** committed to the repo (encrypted)
4. **At build time**, agenix decrypts secrets that the building host has keys for

### Implementation

```
modules/secrets/
├── secrets.nix        # maps secret names → .age files + host public keys
├── aurora-wireguard.age
├── atlas-db-password.age
└── ...
```

`secrets.nix`:
```nix
let
  aurora = "ssh-ed25519 AAAA... aurora";
  atlas  = "ssh-ed25519 AAAA... atlas";
  laptop = "ssh-ed25519 AAAA... laptop";
  all    = [ aurora atlas laptop ];
in {
  "aurora-wireguard.age".publicKeys = [ aurora ];
  "atlas-db-password.age".publicKeys = [ atlas ];
  "shared-secret.age".publicKeys = all;
}
```

### Key concern: Impermanence + SSH host keys
Since `/etc/ssh` is wiped on reboot, we must persist the host SSH keys via impermanence
so agenix can continue to decrypt secrets after reboot:
```nix
environment.persistence."/persist" = {
  directories = [ "/etc/ssh" ];
};
```

### Workflow
- `cd secrets && agenix -e secret-name.age` to edit/create secrets
- Secrets are decrypted on the target host at activation time

## 6. Cachix + GitHub Actions (Remote Builds)

### Problem
Some hosts (e.g. small ARM boards, cheap VPSes) can't build their own NixOS configs.

### Solution
GitHub Actions builds all host configs and pushes closures to cachix. Hosts then
`nixos-rebuild switch --option substituters https://cache.nix-community.org https://your-cache.cachix.org`

### GitHub Actions Workflow

```yaml
# .github/workflows/build-and-push.yml
name: Build and Push to Cachix
on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        host: [aurora, atlas, laptop]
    steps:
      - uses: actions/checkout@v4
      - uses: cachix/install-nix-action@v30
        with:
          extra_nix_config: |
            extra-substituters = https://your-cache.cachix.org
            extra-trusted-public-keys = <your-cache-pub-key>
      - uses: cachix/cachix-action@v15
        with:
          name: your-cache-name
          authToken: '${{ secrets.CACHIX_AUTH_TOKEN }}'
      - name: Build host
        run: nix build .#nixosConfigurations.${{ matrix.host }}.config.system.build.toplevel
```

### Secrets for GitHub Actions
- `CACHIX_AUTH_TOKEN` stored as GitHub repo secret
- No Nix signing keys needed — cachix handles it

### Host-side configuration
```nix
nix.settings = {
  substituters = [ "https://your-cache.cachix.org" ];
  trusted-public-keys = [ "your-cache.cachix.org-1:..." ];
};
```

## 7. Debian → NixOS Migration (nixos-anywhere)

### Strategy

For each Debian server to convert:

#### Phase 1: Preparation
1. **Inventory the Debian server**: document services, configs, data that needs preserving
2. **Create a host aspect** in this repo for the server
3. **Create a disko config** for the target disk layout
4. **Add the host's SSH host key** to `secrets.nix` (gather existing key or plan for a new one)
5. **Back up all important data** off-server

#### Phase 2: Dry-run in VM
1. Use `vm.nix` pattern to spin up a local VM with the same config
2. Verify everything works: impermanence, secrets, services

#### Phase 3: Install via nixos-anywhere
```bash
nix run github:nix-community/nixos-anywhere \
  -- --flake .#aurora root@aurora-ip
```

This will:
1. SSH into the Debian machine
2. kexec into a NixOS installer
3. Run disko to partition/format disks
4. Install the NixOS closure
5. Reboot into NixOS

#### Phase 4: Post-install
1. Restore backed-up data to `/persist`
2. Re-key secrets if SSH host key changed: `agenix --rekey`
3. Verify all services are running

### Important notes
- **This is destructive** — the Debian system is completely replaced
- You need SSH access to the Debian machine with root/elevated privileges
- kexec must be available (standard on most Debian kernels)
- For Hetzner servers: use the Hetzner rescue system instead of kexec for reliability
